cellranger-aws-pipeline

https://github.com/melaniedavila/cellranger-aws-pipeline · Autores: 14 estrellas · Python

¿Qué es y para qué sirve?

cellranger-aws-pipeline es una implementación en la nube de Cell Ranger, el software de 10x Genomics para el análisis de datos de secuenciación de RNA de célula única (scRNA-seq). Este pipeline permite ejecutar el procesamiento de datos de scRNA-seq en Amazon Web Services (AWS), aprovechando los recursos escalables de la nube para acelerar el análisis y reducir el tiempo de ejecución local.

Instalación

El pipeline se implementa principalmente mediante Python y requiere el uso de pip para instalar sus dependencias. Además, se necesitan permisos y configuraciones en AWS (como credenciales y roles). Sigue estos pasos: 

pip install cellranger-aws-pipeline

Nota: Asegúrate de tener instalado y configurado awscli con credenciales válidas en tu entorno antes de ejecutar el pipeline. También se recomienda usar un entorno virtual para evitar conflictos de dependencias.

Uso básico

A continuación se muestra un ejemplo ilustrativo de cómo iniciar el pipeline. La sintaxis real puede variar según la versión del software y los parámetros específicos del proyecto. 

cellranger-aws-pipeline --input-dir /path/to/inputs --output-dir /path/to/output --job-name my-scRNA-seq-job --region us-east-1

Nota: Esta sintaxis es ilustrativa. Verifica los comandos exactos en la documentación oficial del proyecto, ya que el nombre y los flags pueden cambiar según la versión del pipeline.

Flujo de trabajo típico

El flujo de trabajo típico con cellranger-aws-pipeline incluye los siguientes pasos:

  1. Preparar los archivos de entrada (FASTQ) generados por el secuenciador.
  2. Subir los datos a un bucket de Amazon S3.
  3. Ejecutar el pipeline en AWS usando la línea de comandos.
  4. Monitorear el progreso mediante AWS Step Functions o CloudWatch.
  5. Descargar los resultados (contenedores de análisis, matrices, archivos de expresión) desde S3.
  6. Realizar análisis posteriores en R o Python (por ejemplo, con Seurat o Scanpy).

Errores comunes y buenas prácticas

  • Problemas de permisos en AWS: Asegúrate de que las credenciales de AWS estén configuradas correctamente con aws configure y que el rol tenga permisos para S3, Lambda, ECS, y Step Functions.
  • Archivos de entrada mal formados: Valida tus archivos FASTQ antes de subirlos a S3. Usa herramientas como fastqc para verificar el control de calidad.
  • Uso ineficiente de recursos: Elige una instancia de AWS adecuada (por ejemplo, g4dn.xlarge o c5.4xlarge) según el tamaño del conjunto de datos.
  • No usar entornos virtuales: Instala el pipeline en un entorno virtual para evitar conflictos de dependencias.
  • Ignorar la facturación: Monitorea constantemente el uso de recursos en AWS para evitar costos inesperados.
